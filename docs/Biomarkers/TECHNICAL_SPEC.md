# Техническое задание: Heracles AI Coach

## 1. Общие сведения

### 1.1 Наименование продукта
Heracles AI Coach — цифровой коучинговый модуль внутри платформы Heracles.

### 1.2 Назначение
Предоставление пользователям платформы Heracles персонализированного велнес-коучинга по питанию, тренировкам, сну и восстановлению на основе AI (LLM), интегрированного с данными биомаркеров, носимых устройств и профиля пользователя.

### 1.3 Целевая аудитория
Мужчины, заинтересованные в улучшении здоровья и физической формы, пользователи платформы Heracles, имеющие данные биомаркеров крови и/или носимых устройств.

### 1.4 Правовое позиционирование
Продукт является **велнес-коучем**, а не медицинским устройством. Коуч предоставляет общее образование в области питания и тренировок, персонализированное данными пользователя. Коуч **не ставит диагнозов**, **не назначает лечение**, **не интерпретирует результаты клинически**.

## 2. Техническая архитектура

### 2.1 LLM-интеграция

**Требования:**
- Модель-агностичная архитектура: поддержка Claude (Anthropic) и GPT-4 (OpenAI) с возможностью переключения без изменения кода.
- Абстракция LLM-вызова за сервисным слоем с единым интерфейсом.
- API-ключи хранятся как переменные окружения, никогда не на клиенте.
- Системный промпт хранится как отдельный конфигурационный файл, редактируемый без передеплоя.
- Стриминг ответов (word-by-word). Фоллбек на индикатор набора текста.
- Graceful обработка rate limiting, ошибок, таймаутов.
- Логирование API-затрат per user per day.
- Модель по умолчанию для разработки: `claude-sonnet-4-20250514`.

### 2.2 Инъекция контекста

На каждом повороте разговора в контекст LLM инжектируются данные пользователя:

| Данные | Источник | Частота обновления |
|--------|----------|-------------------|
| Профиль пользователя (имя, возраст, цели, диет. требования, аллергии, уровень тренированности, лайфстайл) | API Gabriel | На старте разговора; обновление если >5 мин |
| LLM-поддерживаемый коучинговый профиль | Хранилище Denis | Каждый поворот |
| Последние результаты биомаркеров | API Gabriel | На старте разговора |
| История биомаркеров (для анализа трендов) | API Gabriel | На старте разговора |
| Биологический возраст + PhenoAge-разбивка | Расчёт Denis | На старте разговора |
| Heracles Composite Health Score (0–100, 5 доменов) | API Gabriel | На старте разговора |
| Данные носимых устройств: 7 дней + 30-дневные тренды (RHR, HRV, сон, нагрузка, восстановление, активность) | API Gabriel | На старте разговора; обновление если >5 мин |
| Логи еды за последние 7 дней | Хранилище Denis | Каждый поворот |
| Сводка активного плана питания | Хранилище Denis | На старте разговора |
| Сводка активной тренировочной программы | Хранилище Denis | На старте разговора |
| История разговора (последние 20–30 сообщений) | Хранилище Denis | Каждый поворот |

**Кеширование:** не обращаться к эндпоинтам Gabriel на каждое сообщение. Загрузка данных на старте разговора, периодическое обновление или по явному триггеру.

## 3. Система профиля и памяти пользователя

### 3.1 Механизм

1. На каждом разговоре текущий профиль инжектируется в системный промпт.
2. По завершении разговора (или периодически в длинных) — отдельный LLM-вызов для обновления профиля на основе транскрипта.
3. Обновлённый профиль перезаписывает предыдущую версию. Ведётся **история версий** для отладки.

### 3.2 Структура профиля

| Секция | Содержание |
|--------|-----------|
| Идентичность | Имя, возраст, локация |
| Цели | Основные и вторичные цели, мотивация |
| Диетпрофиль | Требования, аллергии, предпочтения, умение готовить, количество приёмов пищи, любимая/нелюбимая еда |
| Тренировочный профиль | Уровень, частота, оборудование, травмы/ограничения, предпочтения, текущая фаза программы |
| Лайфстайл | Режим сна/бодрствования, стресс, алкоголь, паттерны работы, тайминг еды |
| Текущий прогресс | Что работает, что нет, достижения, паттерны приверженности |
| Коучинговые заметки | Поведенческие наблюдения, мотиваторы, коммуникационные предпочтения, темы для follow-up |
| Контекст биомаркеров | Ключевые находки, биологический возраст, тренды, приоритетные маркеры |
| Статус планов | Активный план питания, тренировочная программа, текущая фаза, недавние корректировки |
| Тренды носимых устройств | Недавние паттерны |

### 3.3 Персистентность разговоров

- **Продолжение:** возврат к последнему разговору, загрузка полной истории сообщений, последние 20–30 в контексте LLM.
- **Новый разговор:** свежий чат, но профиль пользователя по-прежнему инжектируется.
- **Список разговоров:** дата, превью первого сообщения, количество сообщений.
- **Удаление:** пользователь может удалить отдельные разговоры.

### 4.2 Полноэкранная вкладка

- Полноэкранный чат.

- Стриминг текста или индикатор набора.
- Коуч использует имя пользователя естественным образом.
- Thumbs up/down на каждый ответ коуча с опциональным текстовым отзывом. Отзыв привязан к конкретному сообщению и разговору.
- Дисклеймер при первом использовании (одноразовый, закрываемый): «Ваш коуч Heracles предоставляет персонализированный велнес-коучинг. Это не замена медицинской консультации.»
- Постоянная ссылка на полный дисклеймер в настройках коуча.

### 4.4 Предложенные вопросы

3–4 тапабельных предложенных вопроса под полем ввода, ротируемые по стадии пути пользователя:

## 5. Онбординг

### 5.1 Формат
Разговорный онбординг (не форма). Коуч задаёт вопросы естественно, по 1–2 за раз. Вся информация сохраняется в профиль.

### 5.2 Этапы онбординга

1. **Знакомство и раппорт** — приветствие, объяснение роли коуча.
2. **Цели и мотивация** — что пользователь хочет достичь и почему.
3. **Текущие привычки питания** — как ест, сколько раз в день, готовит ли, аллергии, предпочтения.
4. **Тренировочная история** — тренируется ли, что делает, как часто, оборудование, травмы.
5. **Лайфстайл** — режим сна, стресс, алкоголь.
6. **Подключение носимого устройства** — какое устройство есть, предложение подключить, объяснение пользы.
7. **Вес, рост, базовые измерения** — для расчёта калорий и макронутриентов.
8. **Темп коучинга** — наблюдение первую неделю или сразу план.

### 5.3 Контекстно-зависимый онбординг
Если у пользователя уже есть данные (биомаркеры, носимое устройство, профиль), коуч **ссылается на них** и **не переспрашивает** уже известную информацию.

## 6. Логирование приёмов пищи

### 6.1 Функциональные требования

- **Кнопка Log Meal:** доступна с дашборда и из чата.
- **Поля:** тип приёма пищи (завтрак/обед/ужин/перекус), свободное текстовое описание, опциональное фото, таймстамп.
- **Разговорное логирование:** пользователь пишет в чат «Только что съел куриный салат на обед» → коуч парсит, подтверждает детали, создаёт запись.
- **Уточнение порции:** после текстового логирования коуч спрашивает «Это была маленькая, средняя или большая порция?».
- **Дневной вид:** все приёмы пищи за сегодня, просмотр по дням.
- **Недельный вид:** визуальный индикатор консистентности логирования (точки/галочки по дням).
- **Оценка макронутриентов:** LLM-оценка калорий, белков, углеводов, жиров. Чётко маркировано как «оценка».
- **Дневные итоги:** сумма всех логов против целей пользователя.
- **Обратная связь коуча:** разговорный, образовательный, без осуждения.
- **Коуч использует логи:** последние 7 дней в контексте, замечает паттерны, связывает с данными носимых устройств/биомаркерами.

## 7. Структурированные выходные данные

Коуч генерирует артефакты в виде **структурированного JSON**, который фронтенд рендерит как интерактивные компоненты.

### 7.1 Планы питания

- Структура на 7 дней, количество приёмов пищи по предпочтениям пользователя.
- Per meal: название, описание, макросы (ккал, белок, углеводы, жиры), время приготовления.
- Раскрываемые рецепты: ингредиенты с количеством и пошаговые инструкции.
- Кнопка **Swap**: замена приёма пищи на нутрициологически аналогичный.
- Персонализация: калорийные/макро-цели, предпочтения, диет. требования, умение готовить, доступное время, сезонность, обратная связь по предыдущим планам.
- Один активный план. Новый заменяет текущий.
- Feedback loop: коуч спрашивает о самочувствии после еды.

**JSON-схема (минимальные поля):**
- Top level: `plan_name`, `date_generated`, `days_count` (7), `target_macros` (daily kcal, protein, carbs, fat).
- Per day: `day_number`, `meals[]`.
- Per meal: `meal_id` (для swap), `meal_type`, `name`, `description`, `macros`, `prep_time_minutes`, `recipe`.
- Per recipe: `ingredients[]` (name, quantity, unit), `steps[]` (step_number, instruction).

### 7.2 Генерация рецептов ad-hoc

По запросу пользователя («У меня есть курица, батат и шпинат — что приготовить?») — рецепт, вписывающийся в текущие нутрициологические цели.

### 7.3 Списки покупок

- Автогенерация из активного плана питания (все ингредиенты за неделю).
- Категоризация: овощи/фрукты, белок, молочные, кладовая и т.д.
- Чекбоксы на каждый пункт, процент выполнения.
- Привязка к дашборду как отдельная карточка. Ручное добавление пунктов.
- Перегенерация при смене плана питания.

**JSON-схема (минимальные поля):**
- Top level: `linked_meal_plan_id`, `date_generated`, `total_items`.
- Per category: `category_name`, `items[]`.
- Per item: `name`, `quantity`, `unit`, `checked` (boolean, default false).

### 7.4 Тренировочные программы

- Недельное расписание: название/тип сессии (например, «Upper Body Strength», «Zone 2 Cardio», «Active Recovery»).
- Per exercise: название, подходы, повторения (или длительность), отдых, краткая заметка по технике.
- Силовые: прогрессивная перегрузка, составные движения, программирование по уровню.
- Кардио: целевая зона ЧСС, длительность, интенсивность. Цель: 150+ мин зоны 2/нед.
- Периодизация: 4–6 недельные мезоциклы с объяснением фаз.
- Прогрессивное масштабирование.
- Один активный план.

**JSON-схема (минимальные поля):**
- Top level: `programme_name`, `phase_name`, `phase_duration_weeks`, `sessions_per_week`, `target_zone2_minutes`.
- Per session: `day`, `session_name`, `session_type`, `exercises[]`.
- Per exercise: `name`, `sets`, `reps` (или `duration`), `rest_seconds`, `form_note`, `target_hr_zone` (только кардио).

### 7.5 Тайминг питания относительно тренировок

Практические рекомендации по пред- и пост-тренировочному питанию, встроенные в поведение системного промпта.

### 7.6 Интеграция с дашбордом

- Все артефакты сохраняемые/прикрепляемые к дашборду (Denis предоставляет данные, Gabriel рендерит).
- Доступ без открытия коуча.
- Обновление в реальном времени при модификации плана в разговоре.
- Визуализация трендов биомаркеров с аннотациями питания/лайфстайла (Denis предоставляет данные, Gabriel рендерит).
- Карточки трендов носимых устройств с однострочным LLM-комментарием (Denis генерирует, Gabriel рендерит).

## 8. PhenoAge / Биологический возраст

### 8.1 Расчёт

- **Методология:** PhenoAge (Levine et al., 2018).
- **Входные данные:** 9 биомаркеров из панели крови + хронологический возраст.

| Биомаркер | Что отражает |
|-----------|-------------|
| Альбумин | Функция печени, нутритивный статус |
| Креатинин | Функция почек |
| Глюкоза (натощак) | Метаболическое здоровье, инсулиновая чувствительность |
| C-реактивный белок (CRP, log) | Системное воспаление |
| Процент лимфоцитов | Иммунная функция |
| Средний объём эритроцитов (MCV) | Размер эритроцитов, нутритивный статус |
| Ширина распределения эритроцитов (RDW) | Вариация размера эритроцитов |
| Щелочная фосфатаза | Метаболизм печени и костей |
| Лейкоциты (WBC) | Иммунная активация, системный стресс |

- **Выходные данные:** биологический возраст (число), разница с хронологическим возрастом, для каждого из 9 маркеров — тянет ли он биологический возраст вверх или вниз.
- **Источник данных:** `GET /api/users/{id}/biomarkers/latest` (Gabriel).
- **Хранение:** `POST /api/users/{id}/biological-age` (Gabriel).
- **Пересчёт:** при поступлении новых результатов биомаркеров.

### 8.2 Поведение коуча

- Объяснение биологического vs хронологического возраста простым языком.
- Постановка целей на основе биологического возраста.
- Отслеживание прогресса при ретестах (3–6 мес).
- Связь ежедневных действий с долгосрочными результатами.
- Выделение приоритетных маркеров и целевых стратегий.
- **Обязательный дисклеймер** при первом представлении результата.
- **ЗАПРЕЩЕНО:** использование языка смертности («mortality risk», «death risk», «life expectancy»). Это явная красная линия.

## 9. Коучинговые поведения (системный промпт)

### 9.1 Коучинг по данным носимых устройств

| Сигнал | Реакция коуча |
|--------|--------------|
| RHR снижается | Поздравить с улучшением кардио-фитнеса, связать с тренировочным поведением |
| HRV стабильно низкий | Предложить фокус на восстановлении — лёгкие сессии, стретчинг, противовоспалительные блюда |
| Сон снижается | Обратить внимание на кофеин, предложить гигиену сна |
| Высокая нагрузка | Пост-тренировочное питание (30г+ белка, сложные углеводы) |
| RHR высокий + низкая активность | Предложить добавить кардио |
| HRV сильный | «Тело хорошо восстанавливается, сегодня хороший день для более тяжёлой сессии» |

### 9.1.1 Проактивный коучинг при синхронизации тренировок

- **Кардио:** подтверждение сессии, прогресс к 150 мин/нед зоны 2.
- **Силовая:** поздравление, обсуждение восстановительного питания.
- **Feedback loop:** после каждой синхронизированной тренировки коуч приглашает к рефлексии.
- **Прогрессивная мотивация:** коуч замечает улучшение со временем.

### 9.2 Образование по биомаркерам

- LDL повышен → omega-3 и растворимая клетчатка. Никогда: «У вас высокий холестерин, вы в группе риска».
- Витамин D низкий → сезонный контекст, диетические источники, обсуждение с врачом.
- Воспалительные маркеры высокие → средиземноморская диета, противовоспалительные продукты.
- Тестостерон снижается → связь со сном, стрессом, композицией тела. Никогда: «Вам следует рассмотреть TRT».
- **Всегда:** «Обсудите с врачом» при обсуждении биомаркеров.

### 9.3 Образование по добавкам

Только добавки с сильной доказательной базой, всегда как образование:
- Витамин D, Omega-3, Креатин моногидрат, Магний, Протеин.
- Никогда: конкретные бренды, дозировки, терапевтическое использование.
- Перенаправление на **консультацию Heracles** (с CTA/ссылкой для бронирования).

### 9.4 Стресс и восстановление

При длительном плохом восстановлении коуч одновременно корректирует:
- Тренировки (снижение интенсивности, восстановительные движения).
- Питание (противовоспалительные продукты, белок, магний).
- Сон (кофеин, экраны, температура, режим).
- Контекст (спросить о самочувствии, адаптировать к ситуации).

### 9.5 Поведенческая психология

| Техника | Применение |
|---------|-----------|
| Самомониторинг | Логирование еды и тренировок как базовая привычка |
| Имплементационные интенции | Конкретные планы: «Добавь яйца к завтраку по будням», а не «ешь больше белка» |
| Прогрессивная постановка целей | Малые изменения, 1–3 за раз |
| Позитивное подкрепление | Празднование побед, без чувства вины при срывах |
| Поддержка автономии | Выбор, а не предписание: «Какой из трёх вариантов вам ближе?» |
| Мотивационное интервью | Эмпатичное исследование амбивалентности |
| Персонализация по имени | Естественное использование имени пользователя |

### 9.6 Антипаттерны (ЗАПРЕЩЕНО)

- Стыд и вина.
- Страх-ориентированные сообщения.
- Нереалистичные ожидания.
- Обещания конкретных результатов.
- Газлайтинг.
- Игнорирование предпочтений пользователя.
- Рекомендация конкретных лекарств или дозировок добавок.
- Диагностика состояний.
- Клиническая интерпретация биомаркеров.

### 9.7 Сезонные корректировки

Адаптация рекомендаций по сезону: витамин D зимой, сезонные продукты в планах питания, корректировка уличных тренировок, учёт колебаний мотивации и энергии.

### 9.8 Работа без носимого устройства

- Коуч полноценно работает без носимого устройства (питание, планы, тренировки, биомаркеры, поведенческий коучинг).
- Периодически предлагает подключить устройство с объяснением пользы.
- При подключении — проактивно приветствует и начинает использовать данные.
- При частичных данных — работает с тем, что есть, не повторяет замечания о пропущенных данных.

### 9.9 Эволюция целей

- Периодическая проверка актуальности целей.
- Плавный переход планов при смене целей.
- Обновление профиля.
- Признание прогресса по предыдущим целям.

### 10.3 Еженедельный снапшот (flow)

1. Фоновая задача Gabriel агрегирует недельные данные пользователя.
2. Данные передаются в AI-слой Denis (внутренний API-вызов).
3. AI-слой передаёт LLM с шаблоном промпта, возвращает саммари.
4. Gabriel сохраняет снапшот и отправляет push-уведомление.

**Содержание снапшота:** консистентность логирования, нутрициологические highlights vs цели, завершённые тренировки vs запланированные, тренды носимых устройств, один actionable insight, одна строка о связи поведения с долгосрочным влиянием на биомаркеры.

**Архив снапшотов:** пользователи могут просматривать предыдущие снапшоты.

### 10.4 Управление уведомлениями пользователем

- Переключение каждого типа уведомлений вкл/выкл.
- Тихие часы (по умолчанию: 21:00–8:00).
- Уровень частоты: Minimal (только Snapshot), Balanced (Snapshot + event-triggered), Active Coaching (все типы).
- 48-часовой snooze при закрытии уведомления.

## 11. Guardrails и безопасность

### 11.1 Guardrails системного промпта

- Никогда не ставить диагноз.
- Никогда не назначать лекарства или конкретные дозировки добавок.
- Никогда не давать клиническую интерпретацию биомаркеров.
- Никогда не использовать страх, стыд, вину.
- Никогда не обещать конкретные результаты.
- Никогда не рекомендовать конкретные бренды добавок.
- Всегда рекомендовать консультацию врача при клинических вопросах.

### 11.2 Обработка сообщений о симптомах и клиническая эскалация

- **Острые симптомы** (боль в груди, головокружение, необъяснимая усталость, затруднённое дыхание) → перенаправление к врачу Heracles / GP. CTA для бронирования.
- **Тревожное/клиническое поведение** (просит диагноз, тревога по результатам) → перенаправление на консультацию Heracles.
- **НЕ** триггерится данными: биомаркеры или скачки носимых устройств НЕ блокируют коуча автоматически. Эскалация — по поведению пользователя.

### 11.3 Флагирование лекарств

При онбординге или при релевантности — промпт: «Я вижу, что вы принимаете [лекарство]. Важно, чтобы ваш врач знал об изменениях в диете.»

### 11.4 Слой валидации выходных данных

Вторичная проверка каждого ответа коуча перед показом. Rule-based pattern matching для MVP:
- Названия лекарств или рекомендации дозировок.
- Диагностический язык («у вас», «это указывает на», «вы страдаете от»).
- Клинический язык о лекарствах.
- Ссылки на лечение, излечение, управление конкретным заболеванием.

Для MVP: **флагирование без блокировки**. Ревью флагов и уточнение системного промпта. При серьёзном паттерне — блокировка.

### 11.4.1 Мониторинг дрейфа персоны

- Детекция: ответы >500 слов без вопроса, неиспользование имени, клинический/формальный язык, отсутствие эмоциональных/ободряющих элементов.
- Мониторинговый дашборд.
- Ремедиация: корректировка системного промпта.

### 11.5 Обработка ментальных кризисов

При раскрытии самоповреждения, суицидальных мыслей, пищевых расстройств:
- Эмпатичный ответ, без минимизации.
- Ресурсы UK: Samaritans (116 123), SHOUT (85258), NHS 111 (option 2), Mind (0300 123 3393), Beat (0808 801 0677).
- Коуч **не продолжает** коучинг в этом разговоре.
- Правила включены в системный промпт и тестируются в adversarial bank.

### 11.6 Приватность данных, GDPR, DUA Act

- Здоровье — «special category data» по GDPR Article 9. Требуется явное согласие.
- UK DUA Act: требования прозрачности автоматизированных решений.
- DPA с Anthropic и OpenAI до запуска (без использования для тренировки моделей, zero-data-retention).
- NHS-уровень безопасности: шифрование at rest и in transit, контроль доступа, аудит-логирование.
- Отдельный consent flow перед первым использованием коуча.
- Полная возможность удаления всех данных коуча по запросу пользователя.

### 11.7 Обратная связь и мониторинг

- Thumbs up/down на каждый ответ.
- Все разговоры логируются и ревьюабельны.
- Флагированные ответы поднимаются для ревью.
- Первые 2 недели после запуска: Tom ревьюит выборку ежедневно.

10–15 реальных разговоров: точность науки, качество планов питания, расчёт макросов, обработка диет. требований.

### 14.3 Клинический ревью языка (Suks/Tet)

10–15 ответов о биомаркерах: образовательный vs клинический язык.

### 14.4 Soft launch

Запуск для малой opt-in группы. Tom ревьюит ежедневно 2 недели. Обновление промпта по мере выявления проблем. Затем расширение.

## 15. Стратегия запуска

1. **Бета-запуск:** малая группа opt-in, бренд «Heracles AI (Beta)», сбор обратной связи.
2. **Итерации:** 2–3 ревизии системного промпта на основе бета-фидбека.
3. **Полный запуск:** все пользователи. Коуч включён в подписку Heracles — без доплаты, без гейтинга, без лимитов сообщений.
4. **Ongoing:** ежемесячный ревью Richie, еженедельный ревью флагов, обновление промпта по необходимости.
